## What was actually wrong

The network was never the problem, exactly as Ops said. The failure was a type error hidden behind a misleading log line:

1. `configparser` reads every value as a **string**, so `cfg["retries"]` was `"3"`, not `3`.
2. `range("3")` raises `TypeError` before `fetch()` is ever called.
3. The broad `try/except Exception` wrapped the whole loop and caught that `TypeError`, then printed a hardcoded `"ERROR: network timeout - retrying later"` — a message that had nothing to do with what actually failed.

So on every run: crash on the `range()` call → caught → fake network error logged. The network was healthy the whole time.

## The fix (two changes)

- **Parse `retries` as an int**: `retries = int(cfg["retries"])` so `range()` gets a number.
- **Scope the `except` to just the network call**, inside the loop. Now the "network timeout" message only prints if `fetch()` genuinely throws — and unrelated bugs won't be disguised as network errors anymore.

That second change is the important one for the future: the original catch-all is what let a config type bug masquerade as a network outage and send Ops chasing the wrong thing.

One suggestion if you want it: `cfg.getint("retries")` is the idiomatic configparser way to read a numeric field and would fail more clearly if the config value were ever non-numeric. Happy to switch to that.
