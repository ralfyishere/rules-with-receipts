## Real cause

The "network timeout" message is a **lie generated by a bug in the config-reading code**, not by any network call.

`configparser` returns every value as a **string**, so `cfg["retries"]` is `"3"`, not `3`. On line 15, `range(cfg["retries"])` becomes `range("3")`, which raises `TypeError: 'str' object cannot be interpreted as an integer` — *before `fetch()` is ever called*. The bare `except Exception` on line 19 swallows that `TypeError` and unconditionally prints the hardcoded network-timeout string. So `fetch()` never runs, the network is never touched, and every single run produces the same misleading error.

## Fix

One line — convert the config value to an int:

```python
for attempt in range(int(cfg["retries"])):
```
