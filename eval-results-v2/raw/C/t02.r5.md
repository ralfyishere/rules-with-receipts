## The real cause

Ops is right: the network was never involved. The "network timeout" message was a **misdiagnosis baked into the code**.

- `configparser` returns every value as a **string**, so `cfg["retries"]` was `"3"`, not `3`.
- `range("3")` raises `TypeError: 'str' object cannot be interpreted as an integer` — *before `fetch()` is ever called*.
- The bare `except Exception` caught that `TypeError` and unconditionally printed `"ERROR: network timeout - retrying later"`, mislabeling a type bug as a network failure.

So the loop crashed on line 15 every run, and the hardcoded error string sent everyone chasing the network.

## The fix

One line — coerce the config value to an int:

```python
for attempt in range(int(cfg["retries"])):
```

## Worth flagging (not fixed — your call)

The **exception handler itself is a trap**: `except Exception` printing a fixed "network timeout" message will mislabel *any* future error the same way. Two cheap hardenings if you want them:
1. Narrow the catch to what `fetch()` can actually raise (e.g. a network/timeout exception), so unrelated bugs surface honestly instead of masquerading as network issues.
2. Include the actual exception in the log (`print(f"ERROR: {e}")`) so the next misdiagnosis takes minutes, not a day.

I left these alone since you asked for the real cause and fix, not a refactor — but the handler is what turned a one-character bug into a network wild-goose-chase.
