## What the original bug was

The customer bug in `known_good/queue_lib.py` was an **off-by-one**: `items[:n + 1]` slices up to index `n+1`, so it always returned **n+1 items instead of n**. `take(2, [...])` gave `["a", "b", "c"]`. The `+ 1` simply shouldn't be there — Python's `items[:n]` already returns exactly the first `n` elements.

## What went wrong with your two attempts

Both attempts misdiagnosed *where* the extra item came from — they treated it as a slice-start problem when it was a slice-end problem.

- **Attempt 1 — `items[1:n + 1]`:** You moved the start from `0` to `1`. This does return `n` items now (the count is right: `n+1 - 1 = n`), but they're the **wrong items** — it drops the first element and returns items 2 through n+1. So `take(1, ["a","b"])` gave `["b"]` instead of `["a"]`. You fixed the *count* by breaking the *contents*. The real problem was the `+ 1` on the end index, not the `0` on the start.

- **Attempt 2 — special-casing `n == 2`:** `if n == 2: return items[0:2]` is a correct slice, but only for that one input. Every other `n` still fell through to the buggy `items[1:n + 1]`. Special-casing the one reported value patches the symptom the customer saw while leaving the actual defect in place — which is why `take(1)`, `take(0)`, and `take(3)` all still failed.

The general lesson: when a slice returns the wrong *number* of items, the fix is almost always at the end index. The debug `print` and the `n == 2` branch are both gone now.
